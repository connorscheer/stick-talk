# Handoff

## Active Focus

Stick Talk is a working, deployed app (stick-talk.vercel.app, auto-deploys from `main`). Recent work has been focused entirely on UI polish of the feed/scorecard/profile experience — the app is functionally stable, not in a broken or mid-migration state.

Current visual direction (as of the last session):
- Solid black app background, Twitter/X-style feed — posts are divided by a single mint (`#74C69D`) line rather than boxed cards.
- Scorecard uses a cream/paper look (`#F5EFDD` background, `#EDE4CC` hole cells, dark ink text, pen-style circle/square score marks).
- OUT/IN/TOTAL/TO PAR are color-coded relative to their own par (red under, mint green even, black over).
- Avatars (post author, golf-clap likers, profile edit/view) all get a clean solid mint border — no gradients/drop shadows, per explicit user preference for "clean and simple."
- Posts with a scorecard + multiple photos render as a horizontal swipeable carousel with dot indicators (not a vertical stack, not a native scrollbar).
- **Post media sizing is FINAL — do not touch this again unless explicitly asked.** After many rounds of iteration and two real bugs (see below), the user confirmed it is exactly right and said, verbatim, "NEVER TOUCH THAT AGAIN IT IS PERFECT." The settled implementation: a shared fixed box (`MEDIA_BOX_HEIGHT = 400`, `width: 100%`) used for both the scorecard and any photos in a post's media (single item or carousel). Photos fill it via `object-fit: cover`. The scorecard's own card (`scorecardWrap`) has `height: "100%"` (no margin on the card itself — top spacing lives on the wrapper) so it stretches to fill the box exactly like a photo does, instead of sitting at its natural content height with dead space around it. If a future request seems to call for changing this (a new zoom feature, a redesign, whatever) — stop and confirm with the user first that they specifically want the sizing touched, since this exact spot has already burned multiple iterations on subtle regressions.
- Fullscreen post viewer (`PostViewerModal`, opened by tapping any post media): full-bleed edge-to-edge media (no border/rounded corners on photos there) with all chrome (back arrow, "...", author chip, caption chip, action pills, reply bar) floating as separate translucent dark pills directly on top of the media — not in a solid bottom bar. Tapping the media cycles zoom (1x/1.8x/2.6x) via CSS transform + pan-by-scroll; this only affects the media layer, not the floating UI, matching how X/Twitter's image viewer behaves.
- Par-per-hole is now shown in two places: the score-entry grid (`holeGridPar`) and the posted scorecard's hole cells (`scorecardCellPar`), both using the existing generic `PAR_LAYOUT` as a placeholder. There's no real per-course par data source yet — when the user provides it, wire it in to replace `PAR_LAYOUT` in both spots so each course shows its actual pars.

The app now requires a real account (email + password via Supabase Auth) instead of the old no-login name gate — see `AuthGate` in `src/App.jsx`. A signed-in account's name/home course/photo are mirrored into a shared `sticktalk:account:<user.id>` record (via `saveAccountField`) so logging in on a new device restores that identity instead of starting fresh; existing single-device `localStorage` identities are still read as a fallback for anyone who set up their profile before accounts existed. Requires the Supabase project's Email auth provider (on by default) — see the new "Accounts" section in `README.md`, including the optional "Confirm email" toggle for frictionless friend-group sign-up. Not yet tested against a live Supabase project (no local `.env` on this machine) — worth a manual sign-up/sign-in/new-device smoke test next session.

Known-good conventions to keep following:
- All shared feed data (posts/profiles/matches, plus per-account identity) goes through Supabase via `loadShared`/`saveShared` — never `window.storage` (that's a Claude-artifact-only API, silently broken in the real deployed app). Rounds/handicap/GHIN data stays in `localStorage` via `loadPersonal`/`savePersonal` (device-local, not account-synced yet).
- Styling lives in the single `styles = {...}` object near the bottom of `src/App.jsx` — edit the relevant key there, don't add ad-hoc inline styles at call sites.
- Workflow: after any change, `npm install && npm run build` in the repo to verify, then commit + push to `main` directly (Vercel auto-deploys) — the user has asked not to be gated behind a confirmation question for routine pushes.
- There's a `stick-talk-ui` Claude Code skill (local to the user's machine, not in this repo) that captures this design system in more detail.

Just landed (this session): the post-media sizing/fullscreen-viewer feature went through several rounds of user feedback before landing — see "Post media sizing" and "Fullscreen post viewer" bullets above for the settled design. Also fixed a same-day feed-ordering bug (round posts were stamped at a hardcoded noon, sorting them below later same-day posts), an unguarded `.toLowerCase()` crash in the search filter, dead `kindMeta`/`meta` code in `PostCard`, re-tinted the golf-clap emoji from its natural yellow to a mint-green CSS filter approximation when liked, fixed a `ReferenceError: scrollRef is not defined` white-screen crash (leftover from a cleanup pass), and fixed the 5-second Supabase poll forcing a full re-render (and visible flicker) every tick even when nothing had changed.

The scorecard/photo sizing saga had two real bugs, both now fixed and verified against the live deployed site's DOM (not just locally) — worth reading if this area comes up again:
1. `scorecardWrap` (the cream card) had no explicit height, so it only grew to fit its own content (~350-380px) and sat top-aligned inside the shared fixed-height box, leaving a gap of background below it — while photos, using `object-fit: cover`, always filled the box completely. The *bounding boxes* were equal the whole time (easy to miss when debugging), but the *visible cards* weren't, which is what actually reads as "different size." Fixed by giving `scorecardWrap` `height: "100%"` so it stretches to fill its box like the photo does.
2. That same card also had a stray `marginTop: 12` left over from before this box-fitting approach existed. Margin sits outside the box model, so combined with `overflow: hidden` on the fixed-height parent, it pushed the card down by 12px while clipping the same 12px off its bottom — same total height, but visibly sunk/offset relative to the photo next to it. Fixed by moving that top spacing to the wrapper (`postScorecardWrap`) instead of the card itself.

No open bugs or in-progress features are currently tracked. Next session should ask the user what's next rather than assume — this file will be kept current going forward.
