- Converted the whole `styles` object (`src/App.jsx`, near the bottom) plus ~40 scattered inline `color="#FFFFFF"`/dark-bg JSX overrides: page/header/nav/modals/inputs/chat bubbles/post viewer all flipped from dark surfaces + white text to white/`#F4F5F1` surfaces + `#171513` text, neutral `#ECE9E0`/`#D8DCD3` borders (with soft shadows) replacing the old mint card borders. Mint stays as the accent (active tab, buttons, chart line, avatar-ring badges, handicap stamp).
- **Explicitly excluded from the flip, per the user's instruction**: every `scorecard*` style key (the cream/paper scorecard component) — zero lines of `Scorecard` touched. It was already light and didn't need to change.
- The header wordmark image (`/stick-talk-wordmark.png`) was white-on-transparent, built for the old dark header — invisible on the new white one. Generated a recolored dark variant (`public/stick-talk-wordmark-dark.png`, same glyph, RGB swapped to `#171513`, alpha preserved) via a one-off PIL script; `StickTalkWordmark` now points at it. Deleted the now-unused white original. The separate onboarding wordmark (`stick-talk-wordmark-signin.png`, still white) is untouched — onboarding stays dark.
- Avatars (`avatarSm`/`postAvatar`) lost their mint ring border and dark `#4A4844` fallback bg (now a light `#EDE9DD` fallback), matching the reference's plain colored-circle look. Decorative mint rings that aren't plain user avatars (`avatarRing` on the profile photo, the handicap-index "stamp" badge, the center nav `+` button) were deliberately left alone — not what the reference was contrasting against.

Removed it entirely (component + its now-dead styles + the `Navigation` icon import) and replaced it with a real **Rounds** tab (`RoundsTab`, bottom nav icon `BarChart3`) showing actual round history.

- Empty state (no rounds logged) shows a "Log a round" button wired to the same `showLogForm` modal the center nav `+` button opens.

### Previous session (2026-07-14) — unchanged this session

### Onboarding carousel (`AuthGate` in `src/App.jsx`)
- The old plain black sign-in screen is now a Strava-style swipeable carousel: wordmark up top, a floating white mockup card per slide (feed post, scorecard, group chat, notifications — all static sample data via `OnboardFeedMock`/`OnboardScorecardMock`/`OnboardGroupsMock`/`OnboardNotifsMock`, not live app state), a headline, dot pagination, then **Join for free** / **Log in** buttons that lead into the actual email+password form (which now has a back chevron to return to the carousel).
- Cards are a fixed 410px tall (`onboardCardFrame`) so all 4 slides are the same size — don't let a future mock's content push that height without adjusting the others to match.
- Slide transitions are animated with **GSAP** (new dependency, `gsap` in `package.json`): direction-aware slide+fade (`power2.out`, ~350-400ms — pulled from the `ui-ux-pro-max` skill's "Page Transition" motion spec), respects `prefers-reduced-motion` (skips the tween, keeps the state change). If GSAP looks like it's not animating, check `document.hidden`/tab focus first — `requestAnimationFrame`-driven tweens (GSAP included) just don't tick in a backgrounded/unfocused tab, which isn't a code bug.
- Background is intentionally a flat black + subtle mint radial glow, not real photos — the user was asked and explicitly chose "clean/gradient" over sourcing stock photography or providing their own photos.

### Signup now collects identity up front
- First name + last name (required) and an optional profile photo (reusing the existing avatar-edit-button/square-crop pattern from the profile tab) are collected directly on the signup form.
- If Supabase returns a session immediately (email confirmation off, the default for this app), the name/photo are saved straight to `sticktalk:account:<user.id>` from `AuthGate` itself. If email confirmation is required, the collected name/photo are stashed in `localStorage` under `sticktalk:pending-signup` and applied automatically the first time that account actually signs in (see the account-loading effect near the top of `App()`) — `NameGate` is now just a fallback for the rare case that record is missing.

### Photo crop tool (feed composer + round-photo attachment)
- Both `ComposerModal` (feed post photos) and `LogRoundModal` (round photo) now show the picked photo inside `PhotoCropBox` — a box that's the *exact* width × `MEDIA_BOX_HEIGHT` (440px) shape `PostMedia` renders it at in the feed, instead of a generic 4:3 preview that didn't match. Drag to reposition, pinch to zoom.
- On post/submit, the current pan/zoom gets baked into the actual saved image via canvas (`bakePhotoCrop`, shared by both flows so they crop identically) — the framing chosen while composing is what's permanently stored, not left up to `object-fit: cover`'s guess.
- **`CROP_BASE_SLACK` (1.15)**: the crop box's baseline zoom starts 15% past pure cover-fit. Without this, a photo whose aspect ratio is close to the box's own ~0.76 (a typical portrait phone photo) had almost no room to drag left/right — read as the drag being "locked." Both `PhotoCropBox` and `bakePhotoCrop` must keep using the same slack factor or the baked image will stop matching what was previewed.
- A `CropFrameOverlay` (corner brackets + rule-of-thirds grid, `pointer-events: none`) sits on top of the photo so the box visually reads as a crop tool, not just a preview.

### Known-good conventions to keep following (unchanged from before)
- All shared feed data goes through Supabase via `loadShared`/`saveShared` — never `window.storage`.
- Styling lives in the single `styles = {...}` object near the bottom of `src/App.jsx`.
- Workflow: `npm install && npm run build` to verify, then commit + push to `main` directly — user has asked not to be gated behind confirmation for routine pushes.
- **This machine still has no real Supabase `.env`.** Every verification this session (carousel slides, signup validation, the crop box's drag/pinch/bake math) was done by writing a throwaway placeholder `.env` for the dev server, driving the UI with synthetic `PointerEvent`/`TouchEvent`/`File`+`DataTransfer` injection via the browser tool's JS exec, and deleting the `.env` again afterward — never committed. A couple of checks needed a temporary `window.__debugMount*` hook appended to the end of `App.jsx` (dev-only, tree-shaken in prod) to mount `ComposerModal`/`LogRoundModal` standalone outside the auth-gated app; always removed before committing. **Next session should still do a real pass against the live site/live Supabase** — actual signup with a photo, actual photo posting on a real phone (touch, not synthetic events), actual round-photo posting.
